# Resume-Tailor v0.3 需求规格

## 1. 目标

将升级版 Skill 文档中的核心能力变成可重复执行、可机器校验的本地流水线：

```text
简历预处理
→ 岗位导入
→ 五维评分
→ Top N 深度分析
→ 可信定制
→ Diff / Changelog
→ 架构图
→ 静态报告
```

项目继续保持本地 Skill 形态，不发展成 Web App、云服务或自动投递系统。

## 2. 事实来源与伦理

候选人事实只能来自：

1. Run 输入快照中的 `resume.md`；
2. Run 输入快照中的 `experience-bank.md`；
3. 用户在当前对话中明确确认的新信息；
4. 用户提供的项目补充 Markdown。

禁止：

- 新增未出现的公司、岗位、项目、技能、证书或工具；
- 编造数字、用户规模、收入、准确率或业务结果；
- 修改身份、学校、公司、职位和时间段；
- 将“参与”升级成“主导”；
- 将待确认信息显示为已确认事实。

缺少数字时使用 `[请填写：...]`，轻度推断使用 `[需确认]`。

## 3. 输入

### 3.1 用户资料

```text
data/resume.md
data/profile.md
data/experience-bank.md
```

### 3.2 岗位

支持：

- 单个或批量 `.jobs.json`；
- 智能体整理后的 JD 文本 JSON；
- 智能体从截图提取的 JSON；
- 浏览器扩展导出的 `resume-tailor-jobs` v1。

### 3.3 项目图补充

可选：

```text
data/projects/<project-id>.md
```

没有补充文件时，只能使用简历和经验库已有信息。

## 4. FR-01 简历预处理

### 4.1 格式检测

检测：

- 至少 2 个 Markdown 章节；
- 至少 2 个列表项；
- 是否存在 5 行以上、每行仅 1-3 字符的异常断行；
- 零宽字符、异常项目符号和连续空行。

格式正常时不得改写 `data/resume.md`。

### 4.2 清洗

只允许修复格式：

- 零宽字符；
- tab；
- 连续空行；
- `·`、`●`、`▪` 项目符号；
- 明确的章节标题；
- 明显的单字/短行断行。

首次实际清洗前保存：

```text
data/resume.raw.md
```

清洗不能改变可见文字语义。

### 4.3 STAR 拆解

输出：

```text
data/resume.star.md
data/resume.md.hash
```

仅处理工作经历和项目经历，跳过个人信息、技能、教育、证书和自我评价。

每个经历块输出 S/T/A/R、技能关键词和缺失提示。无法确认的 S/T/A/R 必须标记
缺失，不得补写。

hash 未变化时复用现有 STAR 文件。

### 4.4 质量审计

Run 输出：

```text
output/<run-id>/resume-audit.json
output/<run-id>/resume-audit.md
```

至少检查：

- 空壳经历；
- 缺少 Action；
- 缺少 Result；
- 数字缺少口径；
- 被动词过多；
- 技能表述过宽；
- `[请填写]` 和 `[需确认]`。

审计只报告，不自动补写事实。

## 5. FR-02 五维岗位评分

### 5.1 维度

每项 0-100：

1. `hard_skills`
2. `experience_depth`
3. `domain_fit`
4. `soft_fit`
5. `job_quality`

v0.3 默认等权，每项 20%。

### 5.2 岗位质量

纯规则检查：

- 技术栈/工具是否具体；
- 职责是否具体；
- 是否有团队或汇报信息；
- 标题与年限是否矛盾；
- 是否出现明显长期招聘、急招泛招或正文模板化信号。

每个扣分必须写入 `analysis.json.job_quality.signals`。

### 5.3 缓存

缓存键：

```text
resume_hash + job_hash + scoring_version
```

三者一致且 `analysis.json` Schema 完整时复用 Layer 1。任一变化必须重算。

## 6. FR-03 Top N 深度分析

默认 `N=5`，可通过配置调整。

排序后前 N 个岗位增加：

- 定制策略表；
- 级别判断；
- JD 薪资原文；
- 市场薪酬状态；
- 岗位可信度；
- 风险和需确认项。

未执行联网检索时必须显示：

```text
市场薪酬：未联网调研
```

不得把 JD 薪资冒充市场薪酬。

## 7. FR-04 可信定制

### 7.1 结构规则

- 身份和履历不可变值冻结；
- 原有顶层章节顺序保持；
- 允许新增一个 `求职概述` 章节，位置固定在标题后；
- 经历块只能在同一章节内排序；
- 不跨章节搬运；
- 不删除经历块；
- 不要求“为了不同而修改”，证据不足时允许正文不变。

这组规则替代旧 Skill 中“禁止新增章节”与“必须新增 Summary”的冲突描述。

### 7.2 HR 开场白

- 中文；
- 不超过 200 个可见字符；
- 不使用虚假年限；
- 不引用 `[请填写]`；
- 输出后由 Validator 检查长度和占位符。

### 7.3 Changelog

Changelog 必须从 `diff.json` 生成。v0.3 展示分类：

- 措辞调整；
- 顺序调整；
- 新增内容；
- 移除/弱化；
- 待回填；
- 待确认；
- 技能板块变化。

每条声明必须绑定 Diff operation ID。

## 8. FR-05 架构与流程图

### 8.1 自动模式

流水线可基于简历和经验库生成已有证据支持的 Mermaid 图。

### 8.2 深挖模式

当用户要求单独画图时：

- 读取可选项目 Markdown；
- 检查起点终点、至少 3 个节点和节点关系；
- 信息不足时生成最多 3 个追问；
- 不足信息写入 `diagram-questions.md`，不自动补齐；
- 支持技术架构和文科业务流程。

v0.3 不要求脚本主动进行多轮聊天；Skill 负责逐轮提问，脚本负责充分性检查。

## 9. FR-06 可选 Cover Letter 与 PDF

### 9.1 Cover Letter

只有用户回答四个问题后，才能生成 `cover-letter.md`：

1. 为什么申请；
2. 最相关经历；
3. 希望解决的问题；
4. 可使用的联系方式和城市。

无回答时只生成问题清单，不生成虚构正文。

### 9.2 PDF

二进制 PDF 依赖可用浏览器/Playwright 或 PDF 工具。v0.3 必须：

- 提供 ATS HTML；
- 检查单栏、标准标题和无隐藏文字；
- 环境缺少渲染器时明确报告“未生成 PDF”。

不得仅因 Skill 文档声明就声称 PDF 已完成。

## 10. FR-07 静态报告

报告必须展示：

- Run 阶段和导入/分析/定制进度；
- 五维评分；
- 总分和推荐等级；
- 匹配证据、缺口、岗位质量信号；
- 定制简历、分析、修改记录、开场白和图；
- Cover Letter 标签仅在文件存在时显示；
- 待确认字段和 Resume Audit 摘要。

保持 `file://` 可用，无服务器依赖。

## 11. FR-08 Skill 包

六个 Skill 必须：

- Frontmatter 名称和目录一致；
- 描述准确覆盖触发条件；
- 不引用不存在的脚本；
- 每个目录提供 `agents/openai.yaml`；
- 主 Skill 只描述已实现能力；
- 详细规则放入 references，避免重复。

## 12. 非功能需求

- Node.js 标准库优先；
- 不引入数据库和常驻进程；
- 所有外部文本进入 HTML 前转义；
- 输入快照不可变；
- 生成结果可通过 Validator 重现；
- 所有新增规则具有自动化测试；
- 个人数据和真实岗位不进入 Git。

## 13. v0.3 验收

必须满足：

1. `npm test` 全部通过；
2. 完整流水线生成 Resume Audit；
3. 每个岗位存在五维评分；
4. Validator 验证五维总分；
5. 相同输入第二次运行命中 Layer 1 缓存；
6. 前 5 岗位包含深度分析；
7. 报告显示五维和 Run 进度；
8. 开场白长度与占位符校验通过；
9. 所有 Skill 不再引用缺失脚本；
10. 六个 Skill 均有 `agents/openai.yaml`；
11. 原 11 岗位数据回归通过。

## 14. 明确延期

- 自动联网薪酬调研；
- 二进制 PDF 稳定渲染；
- Cover Letter 自动批准；
- 浏览器扩展商店发布；
- 自动投递、Tracker、邮件、面试和 Offer。
